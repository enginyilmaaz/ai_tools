1. Temel Kavramlar

Claude Code CLI üç temel yapı taşı üzerine kuruludur. Her birinin farklı bir görevi, çalışma şekli ve context maliyeti vardır.

1.1 Command (Komut)

Tanım: Kullanıcının /komut-adi yazarak tetiklediği giriş noktası prompt şablonları. Mevcut context'e enjekte olur, yeni izole context açmaz.

Konum: .claude/commands/<isim>.md

Frontmatter Alanları

AlanTipZorunluAçıklama
descriptionstringÖnerilenKomutun açıklaması. Autocomplete'de görünür
argument-hintstringHayırArgüman ipucu (ör: [dosya-adi])
allowed-toolsstringHayırİzin istemeden kullanılacak araçlar
modelstringHayırKullanılacak model: haiku, sonnet, opus

Dinamik Değişkenler

DeğişkenAçıklama
$ARGUMENTSKomuta geçilen tüm argümanlar
$0, $1, $20-indexed argüman erişimi
${CLAUDE_SESSION_ID}Aktif oturum kimliği
!`shell-komutu`Shell çıktısını prompt'a enjekte eder (çalışma anında)

Örnek: Basit Komut

---
description: Firmware'i derle ve test et
model: haiku
---

Projedeki firmware'i derle ve birim testleri çalıştır.
Sonuçları özetle.

Örnek: Tam Donanımlı Komut

---
description: GitHub issue'yu numarasıyla çöz
argument-hint: [issue-numarasi]
allowed-tools: Read, Edit, Write, Bash(gh *), Bash(npm test *)
model: sonnet
---

GitHub issue $0'ı kodlama standartlarımıza göre çöz.

## Bağlam
- PR diff: !`gh pr diff`
- Issue detayı: !`gh issue view $0`

## Adımlar
1. Issue açıklamasını oku
2. Gereksinimleri anla
3. Düzeltmeyi uygula
4. Testleri yaz
5. Commit oluştur

Oturum: ${CLAUDE_SESSION_ID}

Öncelik Sırası

KonumKapsamÖncelik
.claude/commands/Sadece bu proje1 (En yüksek)
~/.claude/commands/Tüm projeler2
<plugin>/commands/Plugin etkinse3 (En düşük)

1.2 Skill

Tanım: Yeniden kullanılabilir bilgi paketleri ve iş akışları. Mevcut context'e enjekte olur. İhtiyaç duyulduğunda yüklenir (progressive disclosure).

Konum: .claude/skills/<isim>/SKILL.md

Frontmatter Alanları

AlanAçıklama
nameGörünen ad ve /slash-command tanımlayıcı. Yazılmazsa dizin adı kullanılır
descriptionNe yaptığı. Autocomplete ve auto-discovery için
user-invocablefalse yapılırsa / menüsünden gizlenir — sadece agent preloading için
allowed-toolsSkill aktifken izin istemeden kullanılacak araçlar

İki Farklı Skill Pattern'i

Pattern 1: Agent Skill (Preloaded)
Agent'ın skills: alanında tanımlanır. Agent başlarken context'e otomatik enjekte olur. Agent'ın "bildiği şeyler" kategorisindedir.
Pattern 2: Standalone Skill (On-Demand)
/skill-adi yazarak veya Skill() tool'u ile çağrılır. Bağımsız çalışır, hiçbir agent'a bağlı değildir. İhtiyaç olduğunda yüklenir.

Örnek: Standalone Skill

---
name: svg-card-creator
description: >
  Veri ile SVG görsel kart oluşturur. Rapor veya
  dashboard çıktısı gerektiğinde kullan.
user-invocable: true
---

# SVG Kart Oluşturucu

## Kurallar
- Boyut: 800x400px, viewBox kullan
- Font: sistem fontu, embed etme
- Renkler: CSS değişkeni ile tanımla
- Çıktı: output/ dizinine yaz

## Şablon Yapısı
- Başlık: üst 60px, koyu arka plan
- İçerik: orta alan, grid düzeni
- Alt bilgi: tarih ve versiyon

Örnek: Agent Skill (Preloaded — Gizli)

---
name: api-conventions
description: API endpoint yazım kuralları
user-invocable: false
---

# API Kuralları
- REST endpoint: /api/v1/{resource}
- Hata formatı: { error: { code, message } }
- Pagination: cursor-based
- Auth: Bearer token header'da

1.3 Agent (Alt Ajan)

Tanım: Kendi izole context'inde çalışan otonom aktör. Kendine özel araçlar, izinler, model, hafıza ve kimlik sahibidir.

Konum: .claude/agents/<isim>.md

Frontmatter Alanları

AlanTipAçıklama
namestringGörünen ad
descriptionstringNe zaman çağrılacağı. "PROACTIVELY" yazılırsa Claude otomatik çağırır
toolsstringİzin verilen araçlar: Read, Write, Edit, Bash. Yazılmazsa tümü inherit
disallowedToolsstringYasaklı araçlar
modelstringhaiku, sonnet, opus, inherit
permissionModestringacceptEdits, plan, bypassPermissions
maxTurnsintMaksimum tur sayısı
skillslistBaşlangıçta preload edilecek skill'ler
mcpServerslistAgent'a özel MCP server'lar
hooksobjectLifecycle hook'ları (PreToolUse, PostToolUse, Stop)
memorystringKalıcı hafıza: user, project, local
backgroundbooltrue → arka planda çalışır
colorstringCLI çıktı rengi (green, magenta vb.)

Örnek: Code Reviewer Agent

---
name: code-reviewer
description: >
  Kod yazıldıktan sonra kalite, güvenlik ve bakım
  standartlarını kontrol eder. PROACTIVELY çağrılabilir.
tools: Read, Write, Edit, Bash
model: sonnet
memory: user
color: green
---

Sen bir kod incelemecisisin. Kodu incelerken keşfettiğin
pattern'leri, kuralları ve tekrarlayan sorunları
agent hafızana kaydet.

## İnceleme Kontrol Listesi
1. Güvenlik: Hardcoded secret, injection, XSS
2. Performans: N+1 query, gereksiz hesaplama
3. Test: Yeterli kapsam, edge case'ler
4. Stil: Proje kurallarına uygunluk
⚠ Kritik Kural: Agent'lar Arası İletişim
Agent'lar bash komutuyla birbirini çağıramaz. Bunun yerine Agent() tool'unu kullan: 
Agent(
  subagent_type="code-reviewer",
  description="Kodu incele",
  prompt="Son commit'i güvenlik açısından incele",
  model="sonnet"
)
"launch", "run" gibi belirsiz kelimeler kullanma — Claude bash komutu olarak yorumlayabilir.

2. Hiyerarşi ve Karar Akışı

2.1 Günlük Kullanım Dağılımı

%80
Vanilla Claude Code
İyi CLAUDE.md + direkt prompt
→
%15
Command + Skill
Tekrarlayan iş akışları
→
%5
Agent
İzole ağır görevler
Altın Kural
"Vanilla Claude Code, küçük görevlerde herhangi bir workflow'dan daha iyidir." Gereksiz karmaşıklık ekleme. Workflow sadece tekrarlayan veya koordinasyon gerektiren işler için.

2.2 Karar Ağacı

Görev basit mi?
  → Evet → Vanilla Claude Code, hiçbir şey kurma
  → Hayır ↓

Tekrarlayan bir iş akışı mı?
  → Evet → COMMAND yaz (/build, /test, /deploy)
  → Hayır ↓

Birden fazla yerde kullanılan domain bilgisi mi?
  → Evet → SKILL yaz (protocol-spec, coding-patterns)
  → Hayır ↓

Ana context'i kirletmeden izole iş mi?
  → Evet → AGENT tanımla (reviewer, test-runner)
  → Hayır → Vanilla Claude Code

3. Orchestration Workflow

3.1 Command → Agent → Skill Pattern

En güçlü kullanım şekli üç yapıyı katmanlı birleştirmektir:

KatmanRolAçıklama
CommandGiriş noktasıKullanıcıyla etkileşir, agent ve skill'leri koordine eder
AgentVeri çekiciPreloaded skill ile veri toplar, sonucu command'a döner
SkillÇıktı üreticiBağımsız çalışır, agent'tan gelen veriyle çıktı üretir

3.2 Çalışan Örnek: Weather Orchestrator

# Akış
Kullanıcı → /weather-orchestrator (Command)
  ├── Adım 1: AskUser → Celsius mı Fahrenheit mı?
  ├── Adım 2: Agent tool → weather-agent
  │     └── Preloaded Skill: weather-fetcher
  │     └── Open-Meteo API'den sıcaklık çeker
  │     └── Döndürür: temperature=26, unit=Celsius
  ├── Adım 3: Skill tool → weather-svg-creator
  │     ├── Oluşturur: weather.svg
  │     └── Yazar: output.md
  └── Çıktı: SVG kart + özet dosya

Command Dosyası

# .claude/commands/weather-orchestrator.md
---
description: Hava durumu verisi çek ve SVG kart oluştur
model: haiku
---

1. Kullanıcıya sor: Celsius mi Fahrenheit mi?
2. weather-agent'ı çağır (sıcaklık çekmesi için)
3. Dönen veriyi /weather-svg-creator skill'ine ver
4. Oluşan SVG ve özeti kullanıcıya sun

Agent Dosyası (Preloaded Skill ile)

# .claude/agents/weather-agent.md
---
name: weather-agent
description: Hava durumu verisi çeken ajan
tools: Bash(curl *)
model: haiku
skills:
  - weather-fetcher   # Başlangıçta preload edilir
---

Preloaded skill'deki talimatlara göre sıcaklık verisini çek
ve sonucu döndür.

Temel Prensipler

  • Command = orkestratör (kullanıcı etkileşimi + koordinasyon)
  • Agent = veri çekme (preloaded skill ile)
  • Skill = bağımsız çıktı üretme
  • Her bileşenin tek bir sorumluluğu var: Fetch (agent) → Render (skill)

4. Memory (CLAUDE.md) Yönetimi

4.1 Yükleme Mekanizması

TipYüklemeAçıklama
Ancestor (üst dizinler)Başlangıçta otomatikClaude yukarı yürür, tüm CLAUDE.md'leri yükler
Descendant (alt dizinler)Tembel (lazy)Sadece o dizindeki dosyalarla etkileşimde yüklenir
Sibling (kardeş dizinler)ASLAfrontend/ çalışırken backend/CLAUDE.md yüklenmez
GlobalHer zaman~/.claude/CLAUDE.md — tüm projelerde geçerli

4.2 Monorepo Stratejisi

/proje-root/
├── CLAUDE.md # Root — ortak kurallar (her zaman yüklenir)
├── frontend/
│   └── CLAUDE.md # Frontend kuralları (lazy)
├── backend/
│   └── CLAUDE.md # Backend kuralları (lazy)
├── firmware/
│   └── CLAUDE.md # Firmware kuralları (lazy)
└── api/
    └── CLAUDE.md # API kuralları (lazy)
CLAUDE.md Kuralları
  • Dosya başına 200 satırı geçme (ideal: 60 satır)
  • CLAUDE.local.md ile kişisel tercihleri ayır (.gitignore'a ekle)
  • /memory ve /rules komutları hiçbir şeyi garanti etmez — MUST yazmak bile yetmez
  • Bilgiyi CLAUDE.md'ye tıkmak yerine skill'lere böl

5. MCP Server Konfigürasyonu

MCP (Model Context Protocol) server'lar Claude Code'u harici araçlara, veritabanlarına ve API'lere bağlar.

Konfigürasyon Konumları ve Öncelikleri

KonumKapsamÖncelik
Agent frontmatter (mcpServers)O agent'a özel1 (En yüksek)
.mcp.json (repo root)Proje2
~/.claude.jsonKullanıcı (tüm projeler)3

Önerilen MCP Server'lar

ServerPaketİşlev
Context7@upstash/context7-mcpGüncel kütüphane dokümantasyonunu çeker, hallucination'ı önler
Playwright@playwright/mcpBrowser otomasyonu — UI test, screenshot, form
Chrome DevTools—Gerçek Chrome'a bağlanır — console, network, DOM debug
DeepWikideepwiki-mcpGitHub repo'lar için wiki-style dokümantasyon
Excalidraw—Mimari diyagram, flowchart oluşturma

Örnek .mcp.json

{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp"]
    },
    "playwright": {
      "command": "npx",
      "args": ["-y", "@playwright/mcp"]
    }
  }
}
Uyarı: Fazla MCP = Fazla Karmaşıklık
"15 MCP server kurdum, daha fazla = daha iyi diye düşünerek. Sonuçta günlük sadece 4 tanesini kullanıyordum." — r/mcp (682 upvote). Sadece gerçekten kullandıklarını ekle.

8. Agent Memory (Kalıcı Hafıza)

Claude Code v2.1.33'te tanıtıldı. Her agent'a kendi kalıcı hafızasını verir — oturumlar arası öğrenme sağlar.

Memory Scope'ları

ScopeKonumKimler Görür
user~/.claude/agent-memory/<agent>/Sadece sen, tüm projeler
project.claude/agent-memory/<agent>/Tüm takım, bu proje
local.claude/agent-memory.local/<agent>/Sadece sen, bu proje

Mekanizma

  • Başlangıçta MEMORY.md'nin ilk 200 satırı system prompt'a enjekte edilir
  • Read, Write, Edit otomatik etkinleştirilir
  • 200 satırı geçerse agent detayları topic-specific dosyalara taşır
  • Skills = statik bilgi (başlangıçta) ↔ Memory = dinamik bilgi (zamanla birikir)

Örnek: Hafızalı Agent

---
name: api-developer
tools: Read, Write, Edit, Bash
model: sonnet
memory: project
skills:
  - api-conventions
  - error-handling-patterns
---

API endpoint'leri implement et. Preloaded skill'lerdeki
kurallara uy. Çalışırken mimari kararları ve keşfettiğin
pattern'leri hafızana kaydet.

Başlamadan önce hafızanı kontrol et.
~/.claude/agent-memory/api-developer/
├── MEMORY.md # Ana dosya (ilk 200 satır yüklenir)
├── rest-patterns.md # Konu-spesifik
└── error-catalog.md # Konu-spesifik
